# Step 6 — AI Icebreaker Engine

> Personalized, professional networking questions for every person at every table,
> generated by Claude (on GCP Vertex AI), delivered without ever blocking the room.

This document is the durable reference for how icebreakers are generated, kept
reliable, and kept private. For the build spec see
`docs/product/releases/pre-mvp.md` §9; for the realtime mechanics it rides on, see
`docs/design/realtime.md`.

---

## 1. What an icebreaker is

For each person at a table we generate **one** question they can ask **one specific
other person** at that table. The data model captures exactly that:

```
icebreakers(round_id, table_number,
            recipient_attendee_id,   -- who READS and asks the question
            target_attendee_id,      -- who the question is ABOUT
            question_text, generated_at)
```

A table of 4 → 4 icebreakers (one per recipient), each pointed at a chosen
tablemate. This is the "directed" model: Asha is told what to ask Bobby; Bobby is
told what to ask Chitra; and so on.

---

## 2. The one principle: never block the room

> **Publishing a round must return instantly. Icebreakers arrive a moment later.**

The spec is explicit (§Backend constraints): *icebreaker generation MUST NOT block
the UI.* So generation is **asynchronous** and rides the Step-5 realtime doorbell:

1. The organizer publishes → `rounds.py` writes the round + table assignments and
   **returns immediately**.
2. `publish_round` schedules `engine.generate_for_round` as a FastAPI
   **BackgroundTask**.
3. Each `INSERT` into `icebreakers` is itself a realtime event → phones re-fetch
   `/live` → the question "pops in" seconds after the table appears.

There is **no new realtime wiring**: `/live` already returns the icebreaker (null
until ready) and `icebreakers` is already in the `supabase_realtime` publication
with `REPLICA IDENTITY FULL`. **No new migration is required for Step 6.**

```
publish ──> round + assignments (instant 201) ──> phones show TABLE
   └─ background ─> per table: 1 Claude call ─> INSERT icebreakers ─> phones show QUESTION
```

---

## 3. One call per table (cost & latency)

Per spec §9, we make **one Claude call per table**, not per person — ~10 calls per
round instead of ~40. Each call returns a question for everyone at that table.

To keep the LLM from corrupting data, **people are referenced by a stable 1-based
index, never by UUID**. The model returns a JSON array keyed by index; the engine
maps index → `attendee_id` itself. A model that mangles an identifier therefore
cannot write a bad row — at worst that entry is dropped and the person gets a
fallback question.

Output is forced to valid JSON by prefilling the assistant turn with `[`.

---

## 4. Reliability: the room never sees a blank

Every failure mode degrades to the **curated fallback bank** (real, professional
questions in `prompts.py`), per person:

| Situation | Behavior |
|---|---|
| LLM disabled (`ICEBREAKER_ENABLED=false`) | All questions from fallback bank |
| Provider misconfigured (`vertex` but no project id) | Logged as error → fallback bank |
| Call errors or **times out** (`> ICEBREAKER_TIMEOUT_SECONDS`) | Fallback bank for that table |
| Output isn't valid JSON | Fallback bank for that table |
| Model skips a person / gives an invalid entry | Fallback for **that person only** |
| A table has < 2 people | Skipped (a question needs someone to be about) |
| Model returns a paragraph / newlines / huge text | **Sanitized** — collapsed to one line and capped at `MAX_QUESTION_LEN` (280) so a phone never renders a wall of text |
| Model repeats a question the person already got | **Rejected** as a literal repeat → fresh fallback (see §5.1) |

The background task itself is wrapped so it **logs and swallows** any error: a
failed icebreaker batch can never crash a request or stall the event. Phones
simply keep showing the table with `icebreaker: null`, and the organizer can
trigger a refresh.

## 5.1 Variety across rounds (anti-boredom)

People meet across 8–10 rounds — the same person must not keep getting the same
question or they tune out. Three mechanisms, all keyed on a recipient's question
history (`icebreakers` rows for that `recipient_attendee_id`, newest first):

1. **The model is told.** Each person's recent questions (last 3) are included in
   the per-table prompt with an explicit instruction to *ask about something
   different this round*.
2. **Repeats are rejected.** If the model returns a question that normalizes
   (case/whitespace/trailing-punctuation insensitive) to one the person already
   got, it's discarded and that person gets a fallback instead — the LLM can't
   accidentally serve a literal repeat.
3. **The fallback bank rotates.** `fallback_question` rotates the bank by how many
   questions the person has already had and skips any they've already seen — so
   even the no-LLM path doesn't repeat itself until the pool is exhausted.

The **"Generate Another"** refresh rides the same path: the current question is in
the person's history, so the refreshed one is guaranteed to differ.

### 5.1.1 The fallback bank: 100 questions, theme-aware

Because the LLM provider is not yet configured, the fallback bank currently carries
**every** icebreaker the room sees — so it is **100 curated questions**, not a
handful. They live in `prompts.FALLBACK_BANKS`, a dict grouped by **intent**:

| Bucket | Count | Used when the round topic is about… |
|---|---|---|
| `general` | 22 | anything (default); also woven with a light Hyderabad/India thread |
| `building` | 12 | the product / vision / what they're building |
| `fundraising` | 11 | capital, investors, runway |
| `gtm` | 12 | first customers, growth, distribution |
| `team` | 11 | hiring, co-founders, culture |
| `lessons` | 11 | mistakes, advice, changed-my-mind |
| `ask` | 11 | the ask / how the room can help |
| `local` | 10 | the Hyderabad / India ecosystem explicitly |

`bucket_for_theme(theme)` keyword-matches the organizer's **round topic** (the same
`round_topics` entry that already steers the LLM) to a bucket; an unrecognised or
empty topic → `general`. The active pool is **that bucket + `general`** (deduped),
so a themed round leans into its topic while still having ~30+ options for the
rotation to draw on. This closes a gap: before, round topics steered only the LLM;
now they shape the fallback too. The questions are single-`{target}` templates, so
they drop straight into the engine's existing index→name mapping — no engine
contract change. Voice is warm and casual but still work-focused (the same content
guardrails as §7 apply to the human author).

## 5.2 Idempotency, cancel, refresh

- **Idempotent PER TABLE** — `generate_for_round` skips only the tables that
  already have icebreakers and generates the rest. This makes it **self-healing**:
  if a generation dies half-way through the room (or a publish is retried,
  REQ-RT-03), a re-run finishes only the missing tables instead of skipping the
  whole round and leaving tables permanently blank. No double-generation, no
  double-billing.
- **Cancel** — `cancel_round` already deletes a round's icebreakers (REQ-RT-02);
  nothing extra needed. The bad round leaves no trace.
- **Refresh** ("Generate Another Icebreaker") — `POST …/icebreaker/{id}/refresh`
  is **synchronous** (the button waits), regenerates for the attendee's current
  table, and **inserts a new row**. `/live` and `GET …/icebreaker/{id}` already
  return the latest by `generated_at`, so the newest question wins. Falls back to a
  curated question if the LLM is unavailable, so the tap always produces something.

## 6. Architecture (modular, prompts in one place)

```
app/icebreakers/
  prompts.py   ← THE single source of all prompt text: system prompt (guardrails),
                 the per-table user-prompt builder, the JSON prefill, the
                 intent-bucketed fallback banks (100 questions) and the
                 theme→bucket router. No prompt strings exist anywhere else.
  provider.py  ← LLMClient protocol + VertexClaudeClient / StubClient / DisabledClient
                 + get_llm_client() chosen by env. Swap models/providers here only.
  engine.py    ← orchestration: generate_for_round / refresh_for_attendee,
                 JSON parse + validate + index→UUID map, persistence, logging, audit.
```

- `routers/rounds.py` adds **one** line in the publish seam (schedule the task).
- `routers/icebreakers.py` `refresh` calls `engine.refresh_for_attendee`.
- Engine functions take an optional `client` so tests inject fakes; production uses
  `get_llm_client()`.

### Provider selection (`LLM_PROVIDER`)
| Value | Use |
|---|---|
| `vertex` (default) | Claude on GCP Vertex AI — uses GCP credits + Application Default Credentials, **no API key** |
| `stub` | Deterministic offline client — the entire test suite, and local dev with zero GCP setup |
| `disabled` | Never calls the LLM — always the fallback bank |

A `vertex` provider with no `VERTEX_PROJECT_ID` degrades to `disabled` (real
fallback questions), never to stub gibberish in production.

### Configuration (all env-overridable, no magic numbers in code)
```
LLM_PROVIDER=vertex
VERTEX_PROJECT_ID=<your gcp project>
VERTEX_REGION=us-east5
VERTEX_MODEL=claude-sonnet-4-6
ICEBREAKER_ENABLED=true
ICEBREAKER_MAX_TOKENS=1024
ICEBREAKER_TEMPERATURE=0.7
ICEBREAKER_TIMEOUT_SECONDS=8.0
```

## 7. Guardrails & privacy

- **Guardrails** live entirely in `SYSTEM_PROMPT`: professional/work-focused, one
  sentence ending in `?`, under 30 words, and an explicit ban on personal,
  emotional, political, religious, health, or romantic topics.
- **PII boundary**: names, roles, and "looking for" **are** sent to the LLM —
  personalization requires it. They are **never** written to logs or the audit
  trail. Logs carry UUIDs/counts only (`event_id`, `duration_ms`, `source`,
  `count`); audit metadata carries counts only.

---

## 8. Observability

- **Logs** (per table): `event_id`, `duration_ms`, `source` (`llm`|`fallback`),
  `count`. JSON in production (Cloud Logging), text in dev.
- **Audit** (`audit_log`, service-role only):
  - `icebreaker.generated` per round → `{table_count, llm_count, fallback_count}`
  - `icebreaker.refreshed` per tap → `{table_number}`

## 9. GCP / Vertex setup (operator checklist)

1. Pick the GCP project that holds the credits → set `VERTEX_PROJECT_ID`.
2. Enable the **Vertex AI API** on that project.
3. In **Model Garden**, enable/request access to the **Claude Sonnet** model.
4. Region: default `us-east5` (Claude is served there); override `VERTEX_REGION` if
   you prefer another supported region.
5. Auth: locally `gcloud auth application-default login`; on Cloud Run the service
   account is used automatically (grant it the Vertex AI User role).
6. `requirements.txt` pins `anthropic[vertex]` (adds the Vertex client + google-auth).

Until these are set, `LLM_PROVIDER=vertex` with no project id serves the fallback
bank — the app runs and the room still gets questions.

## 11. Out of scope (Pre-MVP)

- Pre-event / intent-based matching (explicitly excluded by the spec).
- Streaming the question token-by-token (it pops in whole).
- Per-attendee model personalization beyond role + "looking for".
- Caching/reusing questions across rounds (each round is fresh).
- **Organizer-editable fallback bank (DB-backed).** Deferred to post-pilot. The
  100 questions live in code (`prompts.FALLBACK_BANKS`) — version-controlled,
  zero-query, can't fail at the live event. A future feature could move them to a
  small `icebreaker_templates` table read+cached at startup so organizers add/edit
  their own questions and tag them by bucket; the `bucket_for_theme` routing and
  `fallback_question` rotation would carry over unchanged. Not built now (feature
  freeze), but the seam is clean for it.
